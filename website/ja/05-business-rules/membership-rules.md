# メンバーシップと請求ルール — 中央メンバーシップ & SSO ハブ

## 1. ドキュメントの目的

このドキュメントは、システムの中心メンバーシップ & SSO ハブで適用する必要がある主要なビジネス ルールをすべて説明します。

このドキュメントは、以下の役割の指針として機能します。

- プロダクト オーナー
- プログラマー
- UI/UX デザイナー
- テスター
- オペレーショナル チーム

プログラマーは、ここで定義されていないルールについての仮定を立てることはできません。

---

# 2. 一般原則

1. 1 つのメンバー アカウントは、複数の SaaS プロダクトに同時にサブスクリプションを購入できます。
2. 各プロダクトは、独立したパッケージと価格を定義します。
3. ライセンスは、有効なアクティベーション (無料または支払いが確認された場合) が実行された後のみ有効になります。
4. メンバー データはプライベートであり、メンバー オーナーとスーパー アドミンだけが有効な理由でアクセスできます。
5. 支払いプロセスは原子的です: ライセンスは支払いが確認されていない場合に有効になりません。
6. メンバー データは、ライセンスのステータスが変化した場合に永久に削除されません。
7. すべての重要なアクションは追跡可能でなければなりません。

---

# 3. メンバー アカウント ルール

## 3.1 登録

1. メール アドレスはシステム全体で一意でなければなりません。
2. メール アドレスは小文字化され、スペースが削除され、保存される前に正規化されます。
3. パスワードはテキストとして保存されず、ハッシュ化されます。
4. アカウントは `unverified` 状態で保留され、メール アドレスが確認されるまでです。
5. `unverified` アカウントはライセンスを有効化または SaaS アプリにログインできません。
6. メール アドレス確認リンクには有効期間があります (例: 24 時間)。
7. メンバーは再送リンクを要求できます。

## 3.2 ログイン

1. ログインはメール アドレスとパスワードを使用します。
2. メール アドレスはログインの際に正規化されます。
3. ログインの失敗は制限されます (レート制限)。
4. `suspended` アカウントはログインできません。
5. `unverified` アカウントは SSO を介して SaaS アプリにログインできません。

## 3.3 プロフィール

1. メンバーは自分のプロフィールのみを変更できます。
2. メール アドレスの変更には新しいメール アドレスへの確認メールが必要です。
3. パスワードの変更はすべてのアクティブ セッションを終了します。

---

# 4. プロダクトとパッケージ ルール

1. 各 SaaS プロダクトは一意のプロダクト コード (例: `NTO`) で識別されます。
2. 各プロダクトは独立したパッケージを定義します。
3. パッケージの価格は IDR で保存されます。
4. 無料パッケージには有効期間がありません。
5. 有料パッケージには有効期間 (日数) があります。
6. スーパー アドミンはパッケージを追加、変更、または非活性化できます。
7. パッケージの価格変更は既存のサブスクリプションに影響しません。
8. 非活性化されたパッケージは新規メンバーによって選択できませんが、既存のサブスクリプションは有効期限まで続きます。

---

# 5. ライセンス ルール

## 5.1 ライセンス作成

1. 1 つのメンバーは 1 つのプロダクトあたり 1 つのライセンスを同時に有効化できます。
2. ライセンス ID は最初にプロダクトを有効化したときに 1 回だけ作成されます。
3. ライセンスの更新はライセンス ID を変更しません。
4. ライセンス ID はライセンスがキャンセルされた場合にのみ再生成されます。
5. ライセンス ID はシステム全体で一意でなければなりません。

## 5.2 ライセンス有効化

1. 無料パッケージの有効化はライセンスを `active` 状態にします。
2. 有料パッケージの有効化は、支払いが確認された Webhook の受信と検証後に実行されます。
3. システムは有効な支払い確認なしで有料パッケージを有効化できません。

## 5.3 ライセンス ID のフォーマット

```
[プロダクト プレフィックス]-[XXXX]-[XXXX]-[XXXX]
```

- プレフィックス: 2-4 桁の大文字のプロダクト コード
- セグメント: 4 桁のアルファ数字の大文字、ランダムに生成されます
- 例: `NTO-A1B2-C3D4-E5F6`

## 5.4 ライセンス ステータス

- `active`: ライセンスが有効
- `grace_period`: 有料、期限切れ、トレーニング期間中
- `suspended`: トレーニング期間が終了し、アクセスがブロックされました
- `cancelled`: キャンセルされました

ステータス遷移:

```
active → grace_period (有料パッケージのみ、期限切れ時に)
grace_period → active (更新が成功した場合)
grace_period → suspended (トレーニング期間が終了した場合)
suspended → active (メンバーが再度支払った場合)
```

# 6. 支払い ルール

## 6.1 オーダー

1. チェックアウト セッションごとに、ステータス `pending_payment` のオーダーが作成されます。
2. オーダーには一意の `order_id` が割り当てられ、支払いゲートウェイへの参照として使用されます。
3. オーダーが一定期間未決着の場合、自動的に期限切れになります。

## 6.2 支払い確認 Webhook

1. システムは支払いゲートウェイから受信した Webhook の確認後にのみライセンスを有効化します。
2. Webhook は支払いゲートウェイから提供される署名検証メカニズムを使用して検証されます。
3. 1 つの `order_id` は Webhook を受信した場合にのみ 1 回だけ有効化されます (idempotency)。
4. 支払いゲートウェイから受信した Webhook が検証できなかった場合、無視され、ログに記録されます。

## 6.3 支払い失敗

1. 支払い失敗はライセンスを有効化しません。
2. メンバーは新しいオーダーを作成して再度チェックアウトできます。

## 6.4 トランザクション履歴

1. 各トランザクションは永久保存されます。
2. メンバーは請求書をダウンロードできます。
3. トランザクション履歴は変更または削除できません。

# 7. トレーニング期間 ルール

1. トレーニング期間は有料パッケージのみに適用されます。
2. トレーニング期間の長さは各プロダクトで構成されます (デフォルト: 7 日)。
3. トレーニング期間は `expired_at` が達成された時点から始まります。
4. トレーニング期間中:
   - メンバーはアプリケーションにアクセスできます。
   - システムはメンバー ダッシュボードとアプリケーション内で警告メッセージを表示します。
5. 指定されたスケジュールに基づいてメール通知が送信されます。
