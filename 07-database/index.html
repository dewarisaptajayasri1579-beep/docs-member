Database Documentation

Dokumen gabungan ERD, spesifikasi tabel, dan data dictionary untuk Central Membership & SSO Hub.

Entity Relationship Diagram (ERD)

Database utama memakai PostgreSQL 15+ dan Prisma. Primary key berupa UUID, semua waktu bertipe timestamptz dalam UTC, dan penamaan menggunakan snake_case.

members ──< licenses >── products ──< plans
   │               │              │
   ├──< orders >──┴──────────────┘
   │       │
   │       └──< payments ──< webhook_events
   ├──< email_verification_tokens
   ├──< password_reset_tokens
   ├──< authorization_codes >── oauth_clients
   ├──< refresh_tokens >── oauth_clients
   ├──< audit_logs
   └──  notification_preferences

jwt_signing_keys ──< refresh_tokens
RelasiKardinalitasAturan integritas
Member → License1 : NSatu member hanya boleh memiliki satu lisensi yang masih berlaku per produk.
Product → Plan1 : NPaket selalu dimiliki tepat oleh satu produk.
License → Order1 : NOrder dapat membuat lisensi baru atau memperpanjang lisensi yang ada.
Order → Payment1 : NMendukung percobaan ulang; hanya pembayaran sukses yang memfinalkan order.
Payment → Webhook Event1 : NPayload/retry gateway dicatat untuk audit dan idempotensi.
OAuth Client → Code/Token1 : NAuthorization code dan refresh token terikat pada client peminta.

Gunakan partial unique index pada licenses (member_id, product_id) untuk status selain cancelled, agar tidak ada lisensi berlaku ganda untuk produk yang sama.

Konvensi umum

  • Primary key: uuid dengan default gen_random_uuid().
  • Nominal: bigint unit minor (Rp50.000 = 50000), bukan floating point.
  • Password, token, dan client secret hanya disimpan sebagai hash.
  • Entitas bisnis utama memiliki created_at dan updated_at.

Kualitas & keamanan data

  1. Foreign key memakai RESTRICT untuk data keuangan/audit dan CASCADE hanya untuk token sementara.
  2. Webhook dengan signature invalid tidak boleh mengubah order, payment, atau license; event tetap dicatat.
  3. Settlement mengubah payment, order, license, dan audit log dalam satu transaksi database.
  4. Akses payload webhook, audit log, serta data OAuth dibatasi untuk Super Admin atau service backend berwenang.
  5. Unique constraint dan idempotency key menangani checkout/webhook ganda; frontend bukan satu-satunya kontrol.
